# Kingdom v2 Design: Decision Log

## Decisions Made

| Question | Decision | Reason |
|---|---|---|
| Q1. How users start work | Chat directly in the manager pane | Most natural; Kingdom stays invisible, and the manager decides when to assign workers |
| Q2. Number of parallel workers | User sets the upper bound; the manager controls dynamically within that limit | Two layers of control; Kingdom acts as the arbiter |
| Q3. Worker-to-worker communication | All communication goes through the manager; no direct worker-to-worker communication | Single coordination point, auditable, saves tokens |
| Q4. Session persistence | B: restore state, user confirms manually before continuing | Avoids unexpected automatic resumption while preserving context continuity |
| Q5. `kingdom up` behavior | Fully automatic, asks only one question (default manager provider) | Lowers the barrier to first use; the system is up and running within 30 seconds |
| Q6. Checker role | Do not introduce a separate checker role; manager review of job results is the check | Reduces protocol complexity; the manager already has a global view and review capability |
| Q7. Provider discovery mechanism | Actively probe at startup (`which`), allow config to override paths; store results in session state; reference dynamically when recommending failover; warn if worker provider is missing, but error if manager provider is missing | Allows startup in a "no worker" state and lowers the barrier; failover will not recommend providers that are not installed |
| Q8. `kingdom swap` CLI | `kingdom swap <worker>` opens a provider picker, `kingdom swap <worker> <provider>` switches directly; first request a worker checkpoint (10-second window), then force the switch on timeout; use the same failover flow | Reuse failover switching logic and avoid two separate implementations |
| Q9. Job visibility | Kingdom only pushes events that require manager intervention (completion, failure, worker request, dependency cleared); the manager calls `workspace.status()` as needed; humans use `kingdom log` for history and the status bar for real-time updates | Reduces noise; the manager does not need full push notifications |
| Q10. Bootstrap details | MCP config includes a `role` field to distinguish manager/worker, and Kingdom uses it to deliver the appropriate tool set; Kingdom automatically injects the standard manager system prompt, and the user can append more in `KINGDOM.md` | When a new provider takes over, the user does not need to restate the role |
| Q11. `kingdom down` | If there are running jobs, ask: [Wait for completion] [Pause and exit] [Force exit]; "Pause and exit" sends each worker a checkpoint request (10-second window) and then stops them one by one; if there are no running jobs, exit immediately; `--force` skips checkpointing and kills immediately | Protects progress while providing an emergency exit path |
| Q12. Multiple workspaces | Each workspace has its own daemon and does not share state; sockets are isolated by a hash of the workspace path | Isolation matters most; multiple processes are acceptable, and this is the simplest implementation |
| Q13. Failover false-positive protection | Heartbeat interval is 30 seconds; trigger failover only after 2 consecutive missed responses (60 seconds); process exit triggers immediately without waiting for a heartbeat; thresholds are adjustable in `config.toml` | Distinguishes hangs from crashes and avoids false triggers caused by network jitter |
| Q14. Direct typing in the worker pane | Allow interaction (keep the emergency escape hatch); when the worker starts, show a static line at the top of the pane saying "Direct input will not be recorded and is only for emergency intervention," and do not repeat it afterward | Do not lock it down; just make the boundary clear to the user |
| Q15. `KINGDOM.md` format | Pure free-form Markdown; Kingdom passes the whole document to the provider; no structured parsing; machine configuration goes in `.kingdom/config.toml`, AI behavior constraints go in `KINGDOM.md` | Constraints are for the AI to read, and free text is clearer than a structured format; separate responsibilities |
| Q16. `job.request()` loop | The manager answers with `job.respond(request_id, answer)`; prefer MCP server-to-client notifications pushed to the worker; if the provider does not support push, fall back to the worker polling `job.request_status(request_id)` every 10 seconds; Kingdom tells the worker which mode to use at startup | Push is cleaner, polling is the fallback for compatibility |
| Q17. File corruption during failover | Do not roll back automatically; let the new provider decide; the handoff brief adds an item saying "The following files were being written when the crash occurred and may be incomplete; please inspect them first" | The AI can judge code state; automatic rollback would discard completed work, and syntax checks cannot catch logical issues |
| Q18. API key management | Kingdom does not store, encrypt, or inject keys; `kingdom up` checks environment variables and gives a clear prompt and setup command if they are missing; the user manages keys themselves | Zero security risk, and compatible with all existing key management approaches (shell profile / direnv / 1Password CLI) |
| Q19. Concurrent worker limit | By default, 3 workers run in the main window; if the manager needs more, additional workers start in a new tmux window; the status bar shows all workers across windows; the limit is configurable in `config.toml` | Keeps the main window clean, scales without a hard cap, and users can switch with standard tmux commands |
| Q20. Notifications when away | Do not send notifications by default; users can configure none / bell / system by event type in `config.toml`; `on_attention_required` should default to bell | Avoids bothering users; they can enable it as needed; bell works cross-platform, while system notifications are implemented separately on macOS/Linux |
| Q21. Git strategy and failover | During failover, the new provider continues on the same branch; checkpointing automatically creates a commit (`[kingdom checkpoint]`) so each checkpoint has a clean diff; `job.complete` does not auto-commit, and the manager decides what happens next | Checkpoint commits solve the accumulating diff problem; the manager decides whether to merge/squash/discard after the job completes |
| Q22. `job.complete` result_summary | Lightly constrained (what was completed, which files changed, optional remaining issues); Kingdom performs basic validation (non-empty, at least 20 words); Kingdom automatically appends the changed files list; no strict structure required | When a job completes, the worker is healthy, so strict constraints are unnecessary; the manager is AI, so free text is enough |
| Q23. `kingdom doctor` | Checks five layers: system dependencies, API keys, daemon status, session connection health, and config files; each issue includes a concrete repair command | This is the first command to run when something is wrong, so it must be actionable, not just an error dump |
| Q24. Provider reconnecting after disconnect | Provider -> Kingdom: exponential backoff starting at 1s, capped at 30s, retry forever; cache tool calls locally during reconnect, and report them after Kingdom recovers; Kingdom -> Provider (after restart): actively re-establish the connection, and mark it offline to trigger failover if there is no response within 15s | Kingdom always comes back, so retry forever; process disappearance is handled by PID monitoring, not the reconnect path |
| Q25. Manager notification mechanism | Notifications are injected into the manager conversation flow as ordinary messages; the manager (AI) sees the message and automatically decides the next step; the user can interrupt at any time; no separate notification channel is used | Conversation is the manager's workspace; a separate channel would require additional implementation from each provider and add complexity |
| Q26. `kingdom up` when a session already exists | If both daemon and session exist: prompt to attach or restart; if the daemon exists but the session is gone: automatically rebuild the session and restore state; if the session name conflicts: report an error; the session name is configurable in `config.toml` (default `kingdom`) | Handle the three cases separately to avoid accidental overwrite |
| Q27. Manager context overflow | Do not compress with summarization; trigger failover directly when the limit is exceeded; the new Manager reconstructs from Kingdom's structured state (job state, `workspace.notes`, action log) and does not depend on conversation history | The Manager's real "memory" is in Kingdom's structured state; conversation history is process, not state; reuse the Manager failover mechanism instead of inventing a new one |
| Q28. Worker idle reuse | Do not reuse; terminate the process directly after idle timeout (default 10 minutes); when the Manager needs a new worker, call `worker.create()` to get a new process and a new context | Cold start cost is very low (seconds), while reusing old context is risky (pollution + earlier context exhaustion) |
| Q29. `subtask.create` flow | A subtask is a job, and the creator is a worker; automatically `depends_on` the creator's job; Kingdom notifies the manager, and the manager decides whether to assign a worker to execute it; a worker cannot assign workers itself | Reuse the job mechanism and do not introduce a new concept; the manager keeps control over all execution |
| Q30. Cascading job cancellation | When canceling job_A, Kingdom checks dependent jobs; if any exist, it asks: [Cancel all] [Keep waiting] [Decide one by one]; do not cascade automatically by default | Cancellation is an active decision, and cascading semantics are different from failure; keep the manager in control of each job |
| Q31. `kingdom log` format | Three views: default job list (status / time / provider / duration); `--detail <job_id>` full timeline for a single job; `--actions` raw action stream (a readable version of `action.jsonl`) | Covers the two "what happened" scenarios: job history and action audit |
| Q32. `kingdom restart` | Restart only the daemon; do not rebuild the tmux session or restart providers; SIGTERM -> 5s -> SIGKILL; the daemon restores state from `.kingdom/` and reconnects to all surviving providers | Suitable for config changes and daemon hot resets; unlike down+up, the session and providers stay uninterrupted |
| Q33. `job.progress` timeout | If there is no update for 30 minutes by default, show a warning (do not fail over automatically); prompt the user with: [Wait] [Request checkpoint] [Trigger switch]; configurable in `config.toml`; independent from heartbeat timeout (60s) | Long-running silence is normal in some cases (for example, running large tests); the user knows best whether the job is truly stuck |
| Q34. Worker naming convention | Format `w{number}` (`w1`, `w2`, etc.); never reused within a session (the same name in logs always refers to the same worker); reset across sessions (when `kingdom up` runs again, start from `w1`); the status bar shows `[Codex:w1]`, and CLI references use `kingdom swap w1` | No reuse within a session keeps auditing clear; reset across sessions prevents numbering from growing without bound |
| Q35. Provider startup arguments | Kingdom includes startup templates for three built-in providers (`claude` / `codex` / `gemini`); the `{mcp_config}` placeholder is replaced automatically; users can override the binary path and args in `config.toml`; custom providers require the user to configure args themselves | v2 only includes the three mainstream providers, while the rest remain open for configuration |
| Q36. `worker.release` semantics | Can only release an idle worker (if a job is running, error out and require cancel first); release = terminate the process immediately, no checkpoint needed; it is the manual early version of idle timeout | An idle worker has no in-progress work, so no graceful stop is needed |
| Q37. `kingdom clean` design | Show what will be cleaned and how much space will be freed, then execute after user confirmation; support `--dry-run` (show only) and `--all` (no time limit); clean three categories: intermediate checkpoints (>7 days), old job results (>90 days), and old action logs (>30 days, compressed into summaries) | Fully disclose before cleaning; do not delete silently |
| Q38. `workspace.note` conflict detection | Do not detect automatically; the Manager reads all notes every time it takes over and decides conflicts and cleanup itself; Kingdom does not run LLM-based detection | The Manager's semantic understanding is more reliable than keyword detection, and this avoids extra LLM cost and latency |
| Q39. `.kingdom/` gitignore | Create `.gitignore` inside `.kingdom/` with the contents `*`; automatically create it during `kingdom up` initialization; do not modify the project root `.gitignore` | Equivalent to gitignore behavior without changing user files |
| Q40. `workspace.notes` persistence | Persist across sessions in `.kingdom/state.json`; do not clear it on `kingdom down`; load it together with job state during `kingdom up` restore; users can manually copy important notes into `KINGDOM.md` for permanence | Notes are dynamically captured constraints and come from a different source than user-authored `KINGDOM.md`, so they should be stored separately |
| Q41. Non-git directories | If a non-git repository is detected, warn and ask whether to continue; automatically downgrade to `strategy = "none"`; checkpoints save only text summaries, with no diff snapshots; do not require git | Some scenarios do not need git (temporary scripts, documentation work); degrading gracefully is friendlier than failing out |
| Q42. Missing `job.result()` | Add it to the Manager tool list; return the full `result_summary` + `changed_files` + checkpoint history + branch name; distinguish it from `job.status()`: status is lightweight and real-time, while result is complete and only meaningful after completion | Fill the gap and complete the Manager toolset |
| Q43. Shared workspace for multiple people | Do not support multiple instances in the same directory; when a second `kingdom up` detects that the socket is already in use, prompt the user to run `kingdom attach`; each person running Kingdom on their own machine is completely isolated (`.kingdom/` is already gitignored) | One workspace, one Kingdom instance, so there is a single control point |
| Q44. Merge `job.create` + `worker.assign` | Add an optional `worker_id` parameter to `job.create`; if provided, assign automatically; if not, keep the job `pending`; do not introduce a new tool or break the existing interface | Preserves the flexibility to separate planning from execution while supporting one-step dispatch |
